3/*
4Virtualization is part of the text editor UI.
5This way, only the text the user can fit in their view (this description is simple and inaccurate), is given to the UI
6in order to be displayed.
7
8The code that goes into calculating the virtualization result, is seemingly CPU intensive enough,
9to warrant that insertion of text by a user should not re-calculate the virtualization result.
10
11A further detail here, moving the scroll position should continue resulting in the re-calculation
12of the virtualization result, this isn't seen as a problem. (albeit the current method of re-calculation
13could be optimized).
14
15A headache inducing thought is dealing with "random access insertion" so to speak.
16If it is presumed a user through the UI and by typing on their keyboard can only edit their virtualization result
17 (because anywhere the cursor goes the scroll position will move to, thus making it the new virtualization result).
18Then how does one handle refactoring, which inserts text above the current virtualization result.
19 Because, this would then move the scrollbar, but in a way that could push the existing virtualization result
20 out of bounds, and result in re-calculation of the virtualization being necessary.
21
22In short, it seems "easy enough" to predict what the virtualization result will be after the user types
23the letter 'a' within the bounds of an existing virtualization result.
24
25But, edits outside of the virtualization result might be a bit more complicated.
26
27Therefore, it is recommended that the "within bounds of an existing virtualization result"
28case is handled firstly.
29
30And that any edit outside of an existing virtualization result would continue to re-trigger the
31virtualization calculation for now.
32
33A concern with this idea is that, if incorrectly implemented there could be 'desync' between
34what the user sees in the text editor, and the actual text value behind the scenes.
35
36It is vital that there never be any 'desync'.
37
38Considering that there exists code to track the 'EditList'.
39
40Perhaps one could track the starting EditIndex, and the ending EditIndex,
41when finalizing an ITextEditorEditContext.
42
43Then, if the 'EditList' entries are seen to be in bounds of the
44'virtualization result' then just insert directly to the 'virtualization result' those edits too.
45
46Else, one could re-calculate the entire virtualization result.
47
48Holding down any key on the keyboard, which inserts a character, demonstrates the current issue with
49re-calculating the virtualization result on every insertion.
50
51In a small enough file, there isn't much performance issues.
52But, after about 10,000 characters it seems to be getting exponentially performing worse and worse
53for every character that is within a file.
54
55If I write a unit test which provides the initial virtualization result, then I can insert text within the bounds
56of said virtualization result with this idea being described.
57
58Afterwards, I can assert that the "modified virtualization result" equals what would have
59been calculated had I re-invoked the calculate virtualization result method.
60
