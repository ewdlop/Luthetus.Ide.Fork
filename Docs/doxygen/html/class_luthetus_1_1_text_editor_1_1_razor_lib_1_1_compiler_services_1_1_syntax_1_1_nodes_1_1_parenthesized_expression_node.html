TypeClauseNode Luthetus.TextEditor.RazorLib.CompilerServices.Syntax.Nodes.ParenthesizedExpressionNode.ResultTypeClauseNode
get

When parsing an expression, there may be a FunctionInvocationNode with a GenericParametersListingNode.

The expression code works however by passing around an 'IExpression expressionPrimary' and an 'IExpression expressionSeconday'.

This means we cannot add common logic for parsing a GenericParametersListingNode unless the expression parsing code were modified, or maybe a call to a separate function to parse GenericParametersListingNode could be done.

But, for now an experiment with making GenericParametersListingNode into a IExpressionNode is being tried out.

Any IExpressionNode that is part of a complete expression, will have the IExpressionNode.ResultTypeClauseNode of Pseudo in order to indicate that the node is part of a expression, but not itself an expression.

Use 'TypeClauseNode IExpressionNode.ResultTypeClauseNode => TypeClauseNodeFacts.Pseudo;' so that some confusion can be avoided since one has to cast it explicitly as an IExpressionNode in order to access the property. (i.e.: GenericParameterEntryNode.TypeClauseNode is not equal to IExpressionNode.ResultTypeClauseNode).
